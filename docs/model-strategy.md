# 모델 전략 — Sonnet 기본, advisor()로 Opus 에스컬레이션

## 기본 모델

| 모델 | 용도 |
|------|------|
| **Sonnet 4.6** | 모든 에이전트·스킬 오케스트레이터 기본값. 코딩·분석·생성 작업에 최적화 |
| **Opus** | `advisor()` 호출 시 자동 라우팅. 복잡한 설계 결정·충돌 해소·최종 품질 검증 |
| **Haiku 4.5** | 단순 검증·분류·포맷팅 경량 작업 전용 (비용 절감) |

## advisor() 호출 절차

`advisor()`는 **현재 대화 컨텍스트 전체**를 Opus에 전달한다.
컨텍스트가 길면 토큰 비용이 급증하므로 **반드시 `/compact` 선행** 후 호출한다.

| 단계 | 명령 | 목적 |
|------|------|------|
| 1 | `/compact` | 대화 컨텍스트 압축 (토큰 절감) |
| 2 | `advisor()` | 압축된 컨텍스트를 Opus에 전달하여 리뷰 요청 |

```
/compact        ← 먼저 실행
advisor()       ← 그 다음 호출
```

## advisor() 호출 시점

| 상황 | 설명 |
|------|------|
| 접근법 결정 전 | 설계 방향 확정 전, 실질적 작업 시작 전 |
| 막혔을 때 | 오류 반복, 수렴하지 않는 결과, 방향 전환 고려 |
| 분석 충돌 | 여러 분석 결과 간 모순 발생, 신뢰 기준 불명확 |
| 완료 직전 | 최종 결과물 저장 직후 품질 검증 요청 |

## 에이전트별 advisor 활용 기준

| 에이전트 | advisor 호출 조건 |
|---------|----------------|
| **analysis-leader** | synthesis-reviewer 결과에서 모순 발견, 해소 방향 결정 필요 시 |
| **build-leader** | 빌드 결과의 검증 기준 충족 여부 불확실, 구조 변경 결정 필요 시 |
| **design-cross-validator** | HUMAN_QUALITY / AI_DETECTABLE 판정이 BORDERLINE 경계선일 때 |
| **prd-validator** | [HIGH] 이슈 발생 후 PRD 재작성 여부 판단 필요 시 |
| **sllm-optimizer** | 10대 최적화 기법 간 상충 관계 발생, 우선순위 결정 어려울 때 |
| **오케스트레이터** | 복잡한 다단계 작업에서 중간 방향 점검 필요 시 |

## Haiku 사용 기준 (경량 작업)

| 에이전트 | 이유 |
|---------|------|
| validator | 구조 검증, 체크리스트 판정 — 단순 패턴 매칭 |
| docs-keeper | 파일 목록 갱신 — 반복적이고 단순한 쓰기 작업 |
| code-validator | artisan 규칙 위반 탐지 — 규칙 기반 검증 |
| search-analyst | 검색 쿼리 생성 및 결과 요약 — 경량 리서치 |
| research-analyst | 시장 조사 요약 — 구조화된 정보 수집 |
| prd-validator | PRD 교차 검증 — 체크리스트 기반 판정 |
