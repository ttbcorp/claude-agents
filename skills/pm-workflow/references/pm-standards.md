# PM 문서 품질 기준

pm-workflow 파이프라인에서 생성되는 모든 문서에 적용되는 품질 표준.

---

## 1. PRD 템플릿

### 필수 섹션 구조

```markdown
# PRD: {제품명}

> 버전: {X.Y}
> 작성일: {YYYY-MM-DD}
> 작성자: {작성자}

## 1. 배경 및 문제 정의
(어떤 문제를 왜 해결해야 하는가. 시장 조사 결과와 유저 페인 포인트 근거 포함)

## 2. 목표
(이 제품으로 달성하려는 비즈니스/유저 목표. 측정 가능한 형태로 작성)

## 3. 스코프
### 3.1 포함 (In Scope)
(이번 범위에 포함되는 기능 목록)

### 3.2 제외 (Out of Scope)
(이번 범위에서 명시적으로 제외하는 기능 및 이유)

## 4. 유저 스토리
(INVEST 기준 충족 필수. 각 스토리에 AC 3개 이상)

### US-{번호}: {스토리 제목}
**As a** {사용자 유형},
**I want to** {행동/기능},
**So that** {기대 결과/가치}.

**수용 기준 (Acceptance Criteria):**
- [ ] AC1: {검증 가능한 조건}
- [ ] AC2: {검증 가능한 조건}
- [ ] AC3: {검증 가능한 조건}

## 5. 비기능 요구사항
### 5.1 성능
### 5.2 보안
### 5.3 접근성

## 6. 성공 지표 (KPI)
(정량적 측정 가능한 지표)

## 7. 가정 및 제약
```

### PRD 필수 조건

| 항목 | 기준 |
|------|------|
| 배경 섹션 | 시장 조사 또는 유저 인터뷰 근거 포함 |
| 목표 | 측정 가능한 수치 또는 조건으로 기술 |
| Out of Scope | 최소 2개 이상 명시 |
| 유저 스토리 | INVEST 기준 모두 충족 |
| 수용 기준(AC) | 스토리당 최소 3개, 검증 가능한 조건으로 작성 |
| 비기능 요구사항 | 성능, 보안, 접근성 중 최소 2개 섹션 |
| 성공 지표 | 정량적 지표 최소 2개 |

### 금지 표현

PRD 작성 시 아래 표현은 모호하므로 사용하지 않는다. 반드시 구체적인 수치나 조건으로 대체한다.

| 금지 표현 | 대체 예시 |
|---------|---------|
| 빠르게 | 응답 시간 2초 이내 |
| 쉽게 | 클릭 3회 이내로 완료 가능 |
| 많은 | 동시 사용자 1,000명 이상 |
| 좋은 UX | System Usability Scale 점수 70점 이상 |
| 안전하게 | OWASP Top 10 기준 취약점 없음 |

## 2. 로드맵 형식

### MoSCoW 우선순위 정의

| 카테고리 | 정의 | 기준 |
|---------|------|------|
| **Must Have** | 반드시 포함해야 하는 핵심 기능 | 없으면 제품 출시 불가 |
| **Should Have** | 중요하지만 초기 출시에 필수는 아닌 기능 | 있으면 큰 가치 제공 |
| **Could Have** | 있으면 좋지만 낮은 우선순위 | 리소스 여유 시 포함 |
| **Won't Have** | 이번 범위에서 제외 | 향후 검토 또는 영구 제외 |

### 로드맵 필수 섹션

```markdown
# 로드맵: {제품명}

> 버전: {X.Y}
> 작성일: {YYYY-MM-DD}
> 기준 PRD: _workspace/02_prd.md v{버전}

## MoSCoW 우선순위

| 기능 | 카테고리 | 우선순위 | 근거 |
|------|---------|---------|------|
| 기능 A | Must Have | P0 | 핵심 가치 제공 |
| 기능 B | Should Have | P1 | 경쟁사 차별화 |
| 기능 C | Could Have | P2 | UX 향상 |
| 기능 D | Won't Have | - | 범위 초과 |

## 기능 의존성

| 기능 | 선행 기능(Predecessor) | 후행 기능(Successor) | 비고 |
|------|----------------------|---------------------|------|
| 기능 B | 기능 A | 기능 C | A 완료 후 착수 가능 |

## 스프린트 계획

### Sprint 1 (1~2주): {스프린트 목표}
**목표:** ...
- [ ] 기능 A (Must Have, 예상 3일)
- [ ] 기능 B (Must Have, 예상 4일)

### Sprint 2 (3~4주): {스프린트 목표}

## 마일스톤

| 마일스톤 | 예상 완료 | 포함 기능 | 완료 기준 |
|---------|---------|---------|---------|
| MVP 출시 | Sprint 2 종료 | Must Have 전체 | ... |
| Beta 출시 | Sprint 4 종료 | Should Have 일부 | ... |

## 리스크 및 가정

| 리스크 | 가능성 | 영향도 | 대응 방안 |
|-------|-------|-------|---------|
```

### 로드맵 필수 조건

| 항목 | 기준 |
|------|------|
| MoSCoW 분류 | 모든 PRD 기능이 분류되어 있음 |
| 의존성 표 | 선행/후행 기능이 명시됨 |
| 스프린트 배치 | Must Have가 초기 스프린트에 배치됨 |
| 마일스톤 | MVP 포함 최소 2개 마일스톤 |
| 스프린트 단위 | 2주 기준, 기능 복잡도에 따라 조정 가능 |

## 3. INVEST 체크리스트

모든 유저 스토리는 아래 6개 기준을 충족해야 한다.

### I — Independent (독립적)

- 스토리가 다른 스토리에 의존하지 않고 독립적으로 개발·테스트 가능해야 한다.
- 점검: "이 스토리만 따로 개발하고 배포할 수 있는가?"
- 위반 예: "기능 A가 완료되어야 이 스토리를 시작할 수 있음" (의존성 제거 또는 분리 필요)

### N — Negotiable (협의 가능)

- 스토리는 구현 방법이 아닌 목표와 가치를 기술해야 한다.
- 점검: "팀과 기능 구현 방법을 자유롭게 논의할 수 있는가?"
- 위반 예: "React 컴포넌트로 버튼을 만든다" (구현 방법이 고정됨)
- 올바른 예: "사용자가 한 번의 클릭으로 주문을 취소할 수 있다"

### V — Valuable (가치 있음)

- 스토리가 유저 또는 비즈니스에 명확한 가치를 제공해야 한다.
- 점검: "이 스토리를 완료했을 때 누가 어떤 이익을 얻는가?"
- 위반 예: 기술 부채 해소만을 목적으로 하는 스토리 (비즈니스 가치 불명확)

### E — Estimable (추정 가능)

- 팀이 스토리의 개발 규모를 추정할 수 있을 만큼 충분히 구체적이어야 한다.
- 점검: "이 스토리의 개발 기간을 대략 추정할 수 있는가?"
- 위반 예: "전체 인증 시스템을 구현한다" (범위 너무 광범위, 분해 필요)

### S — Small (작음)

- 스토리가 한 스프린트(2주) 내에 완료 가능한 크기여야 한다.
- 점검: "이 스토리를 2주 안에 완료할 수 있는가?"
- 위반 예: 2주를 초과하는 스토리 → 더 작은 스토리로 분해 필요

### T — Testable (테스트 가능)

- 수용 기준(AC)이 명확하여 완료 여부를 객관적으로 판단할 수 있어야 한다.
- 점검: "이 스토리의 완료 여부를 누구나 동일하게 판단할 수 있는가?"
- 위반 예: AC가 "사용자가 만족한다" (주관적, 측정 불가)
- 올바른 예: AC가 "로그인 후 대시보드로 3초 이내 이동한다"

### INVEST 판정 기준

| 충족 기준 수 | 판정 | 조치 |
|------------|------|------|
| 6/6 | PASS | 없음 |
| 4~5/6 | WARN | 미충족 항목 개선 권고 |
| 3개 이하 | FAIL | 스토리 재작성 필요 |

## 4. 검증 패턴

### 모순 탐지 패턴

prd-validator가 반드시 확인해야 하는 모순 패턴:

| 패턴 | 탐지 방법 |
|------|---------|
| 목표-스코프 불일치 | 목표에 언급된 기능이 In Scope에 없음 |
| PRD-로드맵 불일치 | PRD 유저 스토리가 로드맵에 미포함 |
| 의존성-순서 충돌 | 의존성 표의 선행 기능이 후행 기능보다 늦게 스케줄됨 |
| Must Have 누락 | Must Have 기능이 초기 2~3 스프린트에 없음 |
| AC 모호성 | AC에 "빠르게", "쉽게", "많은" 등 정량화 불가 표현 포함 |

### 심각도 분류 기준

| 심각도 | 기준 | 예시 |
|-------|------|------|
| [HIGH] | 제품 출시 품질에 직접 영향 | 유저 스토리 AC 없음, PRD-로드맵 불일치, INVEST T 미충족 |
| [MEDIUM] | 품질 저하 또는 개발 혼란 가능성 | AC 3개 미만, INVEST 1~2개 미충족, 모호한 표현 |
| [LOW] | 개선 권장 사항 | 섹션 순서 이탈, 근거 보강 권장 |
