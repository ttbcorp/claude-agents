# sLLM 10대 아키텍처 최적화

소형 LLM(9B급)을 제로 비용 로컬 AI로 전환하는 10가지 아키텍처 기법.
출처: "10 Architectural Optimizations That Turned My 9B Model into a Zero-Cost Task-Completing Local AI"

---

## OPT-01: 구조화 프롬프트 (Structured Prompts)

**효과:** 출력 품질 525% 향상, 속도 36% 개선

산문형 프롬프트 대신 표 형식을 사용한다.

**적용 방법:**
```markdown
❌ 나쁜 예:
"당신은 코드 리뷰어입니다. 코드를 읽고 문제를 찾아서 설명해주세요."

✅ 좋은 예:
| 역할 | 코드 품질 검증기 |
|-----|--------------|
| 입력 | 파일 경로, 언어 |
| 출력 | violations 목록 (JSON) |
| 제약 | 최대 5개 항목, 각 항목 1줄 이내 |
| 예시 출력 | {"violations": [{"code": "BAN-1", "line": 42}]} |
```

소형 모델은 구조화된 지시를 명확하게 따른다. 산문은 해석 오류를 유발한다.

## OPT-02: MicroCompact 도구 결과 (MicroCompact Tool Results)

**효과:** 도구 출력 80~93% 압축, 컨텍스트 창 보존

도구 결과를 최소 필수 정보만 남기고 압축한다.

**적용 방법:**
```python
# 파일 목록 도구 결과 압축 예시
❌ 원본: 전체 파일 메타데이터 (수백 줄)
✅ 압축: "파일: 42개 | 관련: src/auth.ts, src/token.ts | 최신: 2025-04-19"

# 출력 스키마에 명시
출력 최대 토큰: 256
필수 필드만: [name, status, error]
불필요 필드 제거: [timestamp, metadata, raw_output, debug_info]
```

작은 컨텍스트 창에서 불필요한 정보는 핵심 지시를 밀어낸다.

---

## OPT-03: 생산 모드 강제 (Forced Production Mode)

**효과:** 무한 탐색 루프 방지, 작업 완료율 향상

6단계 이후 탐색 금지, 텍스트 출력 모드로 전환을 강제한다.

**적용 방법:**
```markdown
# 에이전트 프롬프트에 명시
생산 모드 규칙:
- 최대 도구 호출: 4회
- 단계 4 이후: 추가 탐색 금지
- 완료 조건: 출력 파일 저장 완료
- 탐색 대신: 현재 정보로 최선의 답 생성

# SKILL.md에서 에이전트 호출 시
"단계 3 이후에는 도구를 사용하지 말고 지금까지 수집한 정보로 결과를 생성하라"
```

소형 모델은 완료 신호를 인식하지 못하면 루프에 빠진다.

## OPT-04: 사고 비활성화 (Disable Thinking Mode)

**효과:** 토큰 소비 8~10배 감소 (1,024 → 131 토큰)

내부 추론 토큰(Chain-of-Thought)을 비활성화한다.

**적용 방법:**
```markdown
# Claude 에이전트 호출 시
Agent(
  model: "haiku",
  prompt: "..."  # think=false — 내부 추론 없이 직접 출력
)

# 에이전트 파일 내 지시
"추론 과정을 출력하지 말 것. 최종 결과만 반환."
"단계별 설명 없이 즉시 스키마 형식으로 출력."

# Ollama 기반 sLLM 호출 시
/set noThink  또는 API 파라미터에서 temperature 낮춤
```

원자 작업은 복잡한 추론이 필요 없다. 추론 토큰은 낭비다.

---

## OPT-05: 지연 도구 로딩 (Deferred ToolSearch Loading)

**효과:** 프롬프트 토큰 339개 절약 (60% 감소)

모든 도구를 초기에 로딩하지 않고 필요한 시점에 동적으로 로딩한다.

**적용 방법:**
```markdown
# 에이전트 파일에 허용 도구 목록 명시
허용 도구: [Read, Write]  ← 필요한 것만
금지 도구: [Bash, WebSearch, WebFetch]  ← 불필요한 것 제외

# SKILL.md에서 도구 로딩 지연
Phase 1에서는 Read만 사용
Phase 2에서만 Write 허용
WebSearch는 Phase 1.5에서만 허용
```

불필요한 도구 스키마가 컨텍스트 창을 차지하지 않도록 한다.

---

## OPT-06: 외부 메모리 메커니즘 (External Memory Mechanisms)

**효과:** 반복 컨텍스트 제거, 세션 간 상태 유지

구조화된 JSON 파일(AutoDream 방식)로 선호도·과거 결과를 저장한다.

**적용 방법:**
```json
// memory.json 기본 구조
{
  "session": {
    "project_root": "/path/to/project",
    "domain": "이메일 자동화",
    "preferences": {},
    "completed_tasks": []
  },
  "context": {
    "last_analysis": "2025-04-19",
    "key_findings": []
  }
}
```

```markdown
# 에이전트 프롬프트에 메모리 로딩 명시
시작 시: skills/sllm-workflow/references/memory.json 읽기
완료 시: memory.json에 결과 기록 후 종료
```

같은 컨텍스트를 매번 재전송하지 않아 토큰을 절약한다.

## OPT-07: KV 캐시 포킹 (KV Cache Forking)

**효과:** 1.1x 가속 (단일 GPU 기준), 특수 백엔드에서 더 높음

동일한 접두사에서 분기되는 여러 경로의 KV 캐시를 공유한다.

**적용 방법:**
```markdown
# 동일 시스템 프롬프트를 가진 에이전트 분기 패턴
시스템 프롬프트 (공유 접두사)
├─ 분기 A: 검증 에이전트
└─ 분기 B: 변환 에이전트

# Ollama 설정 (지원 시)
OLLAMA_KV_CACHE_TYPE=q8_0  ← 캐시 타입 최적화
num_ctx: 2048              ← 컨텍스트 창 제한
```

단일 GPU 환경에서는 효과가 제한적이므로 다른 최적화를 먼저 적용할 것.

---

## OPT-08: 쓰기 검증 규율 (Verified Write Discipline)

**효과:** 하드웨어 오류·권한 문제 감지, 작업 완료율 향상

파일 저장 후 즉시 읽기로 무결성을 확인한다.

**적용 방법:**
```markdown
# 에이전트 파일에 명시
파일 저장 절차:
1. Write({파일 경로}, {내용})
2. Read({파일 경로}) — 첫 줄·마지막 줄 확인
3. 불일치 시 1회 재시도
4. 재시도 실패 시 오류 보고 후 중단

# 검증 조건
성공 조건: Read 결과의 첫 줄 = Write 내용의 첫 줄
```

소형 모델 환경(Raspberry Pi, 저사양 서버)에서 하드웨어 불안정성을 감지한다.

---

## OPT-09: 병렬 부트 파이프라인 (Parallel Boot Pipeline)

**효과:** 시작 시간 9% 단축 (1,189ms → 1,077ms)

독립적인 초기화 단계를 병렬로 실행한다.

**적용 방법:**
```markdown
# SKILL.md에서 병렬 에이전트 실행
❌ 순차 실행:
  1. sllm-validator 실행
  2. 완료 후 sllm-formatter 실행

✅ 병렬 실행:
  동시 실행:
  - sllm-validator (독립적)
  - sllm-formatter (독립적)
  두 에이전트 완료 후 다음 단계 진행

# 독립성 확인 기준
- 입력이 서로 다른 소스
- 출력이 서로 다른 파일
- 상태를 공유하지 않음
```

상호 의존성 없는 에이전트는 항상 병렬로 실행한다.

---

## OPT-10: 안정 시스템 프롬프트 (Stable System Prompt)

**효과:** 동일 요청 처리 시간 182ms → 73ms (60% 단축)

시스템 프롬프트의 불변 부분을 분리하여 캐시를 최대한 활용한다.

**적용 방법:**
```markdown
# 에이전트 파일 구조
## 시스템 프롬프트 (불변 — 캐시 고정)
당신은 {역할}입니다.
출력 형식: {고정 스키마}
규칙: {불변 제약}
--- 여기까지 매 호출 동일 ---

## 사용자 프롬프트 (가변 — 매 호출 변경)
대상: {런타임 변수}
입력 데이터: {실제 데이터}
```

```markdown
# SKILL.md에서 에이전트 재호출 시
동일 에이전트를 반복 호출할 때 시스템 프롬프트 내용을 변경하지 말 것
가변 데이터는 사용자 프롬프트 섹션에만 포함할 것
```

Ollama·vLLM·llama.cpp는 동일한 시스템 프롬프트를 캐시한다.
접두사가 변경되면 캐시가 무효화된다.
